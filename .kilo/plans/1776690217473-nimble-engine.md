# Nx Boundary, Billing, and Release Architecture Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Reorganize the monorepo around clearer Nx boundaries, improve FIRST/SOLID hotspots, and introduce a provider-neutral billing and developer-release architecture for internal apps, developer-facing libraries, and app-as-a-service monetization.

**Architecture:** The repo should move from "Nx as orchestrator" to "Nx as orchestrator plus boundary enforcement." Existing app-heavy business logic should be split into domain-focused libraries, transport adapters should stay in apps/controllers, and developer monetization should be added as a new provider-neutral billing domain that reuses finance tenant scoping while keeping payment-provider specifics in the payments app.

**Tech Stack:** Nx 21, Angular 20, NestJS 11, TypeORM, Jest, Playwright, pnpm, GitHub Actions, Docker/Kubernetes GitOps.

---

## Current-State Findings

1. Nx structure is present, but boundary discipline is weak.
   - `nx.json` uses the standard `apps/` and `libs/` layout.
   - `eslint.config.mjs` disables `@nx/enforce-module-boundaries`.
   - Most `project.json` files use `tags: []`, so there is no usable dependency policy.

2. Several services violate SRP/OCP/DIP enough to block clean monorepo boundaries.
   - `apps/authentication/src/app/app.service.ts` mixes validation, persistence, crypto, MFA, JWT creation, and RPC-shaped response formatting.
   - `apps/gateway/src/controllers/authentication/authentication.controller.ts` mixes HTTP transport with profile creation and permission initialization.
   - `apps/leads-app/src/app/leads.service.ts` is an Angular “god service” that combines lead CRUD, topic discovery, onboarding, resume parsing, and location lookup.
   - `libs/permission-lib/src/lib/permission-builder.ts` and `libs/permission-lib/src/lib/role-init.service.ts` encode app-scope behavior with hard-coded switches and exceptions.

3. FIRST test quality is uneven.
   - `apps/local-hub-e2e/src/app.spec.ts` is slow, environment-coupled, and stateful.
   - Some authentication tests accept multiple failure modes instead of validating one exact behavior.

4. Release and monetization support exist for deployable apps, not external developer packages or SaaS billing.
   - `.github/workflows/build-push.yml` and `.github/workflows/deploy.yml` are image/GitOps oriented.
   - `apps/payments` handles commerce and Lemon Squeezy integration.
   - `apps/finance` already has tenant/appScope concepts that fit developer-account ownership.
   - No provider-neutral billing domain exists for metering, prepaid blocks, entitlements, or package release workflows.

## Target Monorepo Topology

Adopt a tag taxonomy and physical structure like this:

- `type:app` for deployable apps in `apps/*`
- `type:feature` for domain/application orchestration libs
- `type:data-access` for persistence/API client libs
- `type:domain` for core domain logic and policies
- `type:contracts` for DTOs/events/public contracts
- `type:ui` for Angular UI libs
- `type:util` for cross-cutting utilities
- `scope:<domain>` for business domains such as `auth`, `billing`, `finance`, `payments`, `profile`, `leads`, `permissions`
- `platform:server`, `platform:web`, `platform:shared`
- `visibility:internal`, `visibility:publishable`

Recommended ownership rules:

- apps may depend on feature/data-access/domain/contracts/util libs
- feature libs may depend on domain/data-access/contracts/util libs
- domain libs may depend only on contracts/util libs
- contracts libs may depend only on util libs
- UI libs may not depend on server-only data-access libs
- publishable libraries may not depend on app code or broad internal aggregators such as `@optimistic-tanuki/models`

## Recommended Structural Moves

### Nx Boundary and Library Restructure

1. Break up the broad `libs/models` export surface into domain contracts.
   - Keep `libs/models` only as a transitional aggregator if needed.
   - Add targeted libs such as:
     - `libs/auth/contracts`
     - `libs/billing/contracts`
     - `libs/finance/contracts`
     - `libs/leads/contracts`
     - `libs/payments/contracts`
     - `libs/permissions/contracts`

2. Split `libs/permission-lib` by responsibility.
   - `libs/permissions/domain`: role policies, permission policy definitions, app-scope policy registry
   - `libs/permissions/feature-role-init`: orchestration for initialization flows
   - `libs/permissions/contracts`: DTOs and command payload shapes if not already elsewhere

3. Move app-specific orchestration out of gateway controllers.
   - Add `libs/auth/feature-login`, `libs/auth/feature-registration`, or one `libs/auth/feature-account-bootstrap` for profile seeding and permission bootstrapping.

4. Split frontend data access from feature workflows.
   - Replace the current `apps/leads-app/src/app/leads.service.ts` with narrower libs:
     - `libs/leads/data-access`
     - `libs/leads/feature-onboarding`
     - `libs/leads/feature-topics`
     - `libs/leads/feature-flags`

### Billing and Release Architecture

1. Keep `apps/payments` provider-facing.
   - Checkout generation
   - webhooks
   - transaction provider adapters

2. Introduce a provider-neutral billing domain.
   - New app: `apps/billing`
   - New libs:
     - `libs/billing/contracts`
     - `libs/billing/domain`
     - `libs/billing/data-access`
     - `libs/billing/feature-metering`
     - `libs/billing/feature-invoicing`
     - `libs/billing/feature-entitlements`

3. Reuse finance ownership scope.
   - Billing accounts, usage events, and entitlements should always carry `tenantId` and `appScope` using the existing pattern in `apps/finance/src/app/services/finance-scope.ts` and `apps/finance/src/entities/finance-tenant.entity.ts`.

4. Separate internal workspace libs from developer-release packages.
   - Internal-only: current Angular/Nest shared libs
   - Publishable: narrowly-scoped public SDK/contract packages only
   - Candidate publishable packages:
     - `@optimistic-tanuki/billing-contracts`
     - `@optimistic-tanuki/billing-sdk`
     - `@optimistic-tanuki/app-catalog-contracts`
     - `@optimistic-tanuki/entitlements-sdk`

## Risks and Decision Points

1. `@optimistic-tanuki/models` may be deeply coupled across many apps.
   - Use a transitional period where domain-specific contract libs are introduced before deleting the old aggregator.

2. Enabling boundary rules will likely surface many violations quickly.
   - Fix only the high-value domains first; do not attempt a repo-wide rewrite in one PR.

3. The billing domain depends on product strategy decisions.
   - Required decisions:
     - Is the payment provider fixed to Lemon Squeezy, or should Stripe be a near-term second adapter?
     - Is billing account ownership per tenant, per profile, or both?

5. Public SDK packages add compatibility obligations.
   - Add API review and changelog discipline before publish automation is enabled.
   - Prefer narrow packages with stable contract surfaces over broad convenience exports.

4. E2E test stabilization may need seed-data helpers or isolated test tenants.

## Acceptance Criteria

The work is complete when all of the following are true:

1. Nx boundary rules are enabled and enforced for the high-value projects.
2. Authentication, gateway bootstrap, and leads front-end orchestration are split into narrower libs/services.
3. Permission initialization no longer relies on a central app-scope switch plus embedded cross-scope exceptions.
4. FIRST test quality measurably improves in authentication and local-hub E2E.
