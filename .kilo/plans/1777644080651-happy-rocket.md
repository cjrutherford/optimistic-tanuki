# Dev Stack Turnaround and Docs Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Correct the local-stack documentation, make the advertised restart-on-change flow accurate for supported services, and add a faster hybrid inner-loop workflow that avoids full-workspace rebuild latency during day-to-day development.

**Architecture:** Treat the repo as having three distinct local-dev modes: full Compose dev stack, hybrid inner loop, and plain production-like Compose. The implementation should make `docs/devops/docker-compose.md` the single source of truth, add targeted build/watch tooling for scoped iteration, and fix the dev-compose services whose mounts or commands currently prevent restart-on-change from working as documented.

**Tech Stack:** Nx 21, pnpm, Docker Compose, Node/NestJS services, Angular/SSR frontends, nodemon, bash helper scripts, Markdown docs.

---

## Current-State Findings

1. The current full-stack command is expensive by design.
   - `package.json` defines `docker:dev` as `build:docker:dev && docker:build:dev && docker:dev:up`.
   - That means a full Nx build, then Docker image builds, then phased Compose startup happen before the stack is usable.

2. The documented restart-on-change path is narrower than the docs claim.
   - The actual path is `src -> nx build --watch -> dist -> bind mount -> nodemon restart`.
   - Only services with both a `dist` bind mount and a dev command using `nodemon` can restart automatically.

3. At least one advertised service is miswired for dev restarts.
   - `wellness` watches `/app/wellness`, but the dev override does not mount `dist/apps/wellness` there.
   - Rebuilt output therefore never reaches the watched runtime path.

4. Several services in the dev stack do not actually restart on change.
   - `assets`, `videos`, and `video-client` run plain `node ...` commands in the dev override.
   - `video-transcoder-worker` has no repo-wide watch integration and no restart-on-change path.

5. The docs currently duplicate and conflict.
   - `docs/devops/docker-compose.md` is closest to current truth.
   - `docs/development/debugging.md` still describes outdated startup/watch behavior.
   - onboarding docs repeat startup instructions instead of linking back to one canonical workflow page.

6. A faster workflow is already possible in principle.
   - Many apps already expose `serve` targets in `apps/*/project.json`.
   - The quickest inner loop is to keep shared dependencies in Docker and run the actively edited app locally with `nx serve`, rather than relying on full-stack workspace watch mode.

### Task 2: Add Targeted Build/Watch Tooling For Faster Iteration

**Files:**
- Create: `scripts/dev-build-scope.sh`

- Modify: `docs/getting-started-src/developers/workflows.md`

**Step 1: Add a small scoped build/watch helper**

Create `scripts/dev-build-scope.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

WATCH=0
PROJECTS=""

while [ "$#" -gt 0 ]; do
  case "$1" in
    --watch)
      WATCH=1
      ;;
    --projects=*)
      PROJECTS="${1#*=}"
      ;;
    *)
      echo "Unknown argument: $1" >&2
      exit 2
      ;;
  esac
  shift
done

if [ -z "$PROJECTS" ]; then
  echo "Pass --projects=project-a,project-b" >&2
  exit 2
fi

ARGS=(run-many --target=build --projects="$PROJECTS" --configuration=development)

if [ "$WATCH" -eq 1 ]; then
  ARGS+=(--watch)
fi

NX_DAEMON=false NX_ISOLATE_PLUGINS=false pnpm exec nx "${ARGS[@]}"
```

**Step 2: Expose the helper through package scripts**

Add these `package.json` entries:

```json
{
  "scripts": {
    "build:dev:scope": "./scripts/dev-build-scope.sh",
    "watch:build:scope": "./scripts/dev-build-scope.sh --watch"
  }
}
```

Usage examples to document:

```bash
pnpm run build:dev:scope -- --projects=gateway
pnpm run watch:build:scope -- --projects=gateway,authentication
pnpm run watch:build:scope -- --projects=client-interface
```

**Step 3: Document when to use scoped watch instead of full-workspace watch**

Add guidance like this:

```md
Use `watch:build:scope` when you are iterating on one app or a small dependency set.
Use `watch:build` only when you intentionally need the whole Docker dev stack to stay rebuildable.
```

**Step 4: Verify the helper works**

Run:

```bash

bash ./scripts/dev-build-scope.sh --projects=gateway --watch
```

Expected:
- the non-watch command performs a development build for the named project.
- the watch command starts and stays attached.
- no workspace-wide project list is required.

**Step 5: Commit**


1. Prefer the smallest change set that makes the docs true.
   - Do not try to make every single service fully dev-reloadable in the first pass.
   - Fix the services already implied to work, then document the rest accurately.

2. Keep the hybrid workflow generic, not app-specific.
   - A few concrete examples are enough.
   - Avoid adding a package script per app unless a follow-up proves it is necessary.

3. Treat `video-transcoder-worker` as a separate follow-up unless there is a real current need.
   - It is a different runtime model.
   - It should not block the main turnaround improvements.

4. Validate the new workflow with real runtime evidence, not just doc edits.
   - For this task, logs and manual restart confirmation matter more than unit tests.
