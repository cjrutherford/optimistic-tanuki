# Marketing Materials Generator Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Expand `apps/marketing-generator` from a copy-only campaign concept tool into a generator and editor for flyers, brochures, business cards, and web ad assets with structured layouts and Stable Diffusion-backed visual comps.

**Architecture:** Keep the existing concept-generation flow, but extend the request/output models so each concept contains typed, editable marketing materials instead of generic channel snippets. Use the existing Ollama-backed copy enrichment path for copy and layout prompt tuning, and add a separate server-side image render adapter for Stable Diffusion so image generation is isolated behind config and easy to mock in tests. Keep V1 browser editing lightweight: field editing, layout variant switching, image rerendering, and preview/download for generated comps; defer print-grade PDF/CMYK production features.

**Tech Stack:** Angular standalone components, Angular signals, Express SSR server, existing Ollama integration, fetch-based HTTP adapters, Jest component/service tests.

## Current Gaps

The current app is a campaign-copy generator, not a marketing-material generator.

Files proving current scope:
- `apps/marketing-generator/src/app/types.ts`
- `apps/marketing-generator/src/app/services/marketing-generator.service.ts`
- `apps/marketing-generator/src/app/pages/create-page.component.ts`
- `apps/marketing-generator/src/app/pages/results-page.component.ts`
- `apps/marketing-generator/src/app/services/marketing-enrichment.server.ts`
- `apps/marketing-generator/src/server.ts`

Observed gaps:
- `GenerationRequest` has no deliverable type, size, orientation, brand, image, or edit metadata.
- `CampaignAsset` is copy-only and limited to `web`, `email`, `social`.
- The create wizard only collects offer, audience, channel, intent, and tone.
- The results page only previews text blocks and supports full-set regeneration.
- There is no material editor route, no per-asset editing flow, no image generation, and no material-specific export/download.
- Config only supports one Ollama text model connection.

## V1 Scope

V1 includes:
- Generate typed materials for `flyer`, `brochure`, `business-card`, and `web-ad`.
- Attach format presets to each material, including dimensions and surface definitions.
- Generate structured copy blocks and image prompts per material.
- Render image comps through a Stable Diffusion server reachable on the Ollama host.
- Edit copy fields, swap layout variants, rerender images, and regenerate a single material without regenerating the whole set.
- Download generated comp images and copy payloads.

V1 explicitly excludes:
- CMYK/bleed-safe print export.
- Production PDF composition.
- Drag-and-drop freeform layout editing.
- Asset-library persistence beyond current local storage.

## Confirmed External Dependency

The Stable Diffusion service is confirmed to expose an `Automatic1111`-style API. The implementation should still isolate this behind one server adapter, but the initial adapter can directly target a `txt2img` contract rather than waiting on endpoint discovery.

## Notes For The Implementer

Prefer the smallest correct implementation.

Recommended implementation choices:
- Keep `CampaignConcept` as the top-level unit so the existing gallery flow survives.
- Add one new data file for material presets rather than scattering dimensions across components.
- Keep one server adapter for Stable Diffusion so API-shape changes stay localized.
- Keep layout editing form-based in V1. Do not build drag-and-drop or arbitrary coordinate editors yet.
- Since the external image API is confirmed as `Automatic1111`, start with a direct `txt2img` integration and normalize its response into the app’s internal image-slot shape.
